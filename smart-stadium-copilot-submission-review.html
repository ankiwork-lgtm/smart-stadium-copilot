Code Quality — Structure, Readability, Maintainability

A
Single source of truth for types. All shared interfaces live in lib/types.ts with clear doc comments; no type redefinition found across 20+ files.
Separation of concerns. Reasoning layer (lib/gemini.ts), simulation engine (lib/simEngine.ts), and UI components are strictly decoupled. API routes are thin wrappers — no business logic leaks into route handlers.
Consistent naming and style. kebab-case route folders, PascalCase components, camelCase helpers. Every file has a leading JSDoc comment describing its purpose, design references (e.g. "design.md §4"), and task IDs.
Prompt builder is mode-scoped. buildVenueSection() injects only the context each mode needs (e.g. transit data only for transport / translation), reducing token waste and keeping prompts focused.
Duplicate mode arrays. app/api/assistant/route.ts defines VALID_MODES locally as a plain array instead of deriving it from the canonical AssistantMode union in lib/types.ts. Adding a new mode requires updating two places.
Hardcoded venue IDs in fan page. src/app/fan/page.tsx declares GATE_IDS and FACILITY_IDS as static arrays that must stay in sync with data/venue.json. A data-driven derivation would prevent silent drift.
Empty next.config.ts. No security headers (CSP, HSTS, X-Frame-Options) are configured at the framework level. These would strengthen the deployment posture.

Recommended improvements

  1. Derive VALID_MODES in the route handler from the type literal union to eliminate drift.
  2. Fetch venue.json at build time (or use a shared constant module) to drive both the map pin list and the fan-page keyword patterns.
  3. Add HTTP security headers via next.config.ts headers() — at minimum CSP, X-Frame-Options, and Permissions-Policy.

Security — Safe and Responsible Implementation

B+
API key never reaches the client. lib/gemini.ts is server-only and reads GEMINI_API_KEY from process.env. The route layer keeps Gemini calls strictly server-side; no key is bundled into the client JS.
Exhaustive input validation in app/api/assistant/route.ts. All fields are validated against allowlists — mode, role, language, and every item in accessibilityNeeds. A test even confirms that prompt-injection strings in accessibilityNeeds return HTTP 400.
Message length cap. MAX_USER_MESSAGE_CHARS = 2000 prevents context-stuffing / cost amplification attacks from large payloads.
Anti-hallucination grounding. The system prompt hard-codes an exhaustive entity list and explicit refusal instructions ("respond with: I'm sorry, I don't have information…"). This reduces the LLM's ability to be tricked into inventing facilities or routes.
Stream chunk timeout. app/api/assistant/route.ts races each Gemini chunk against a 15 s deadline, preventing a hung Gemini stream from holding server resources indefinitely.
No HTTP security response headers. The app sets X-Content-Type-Options: nosniff on the streaming route but no CSP, X-Frame-Options, or Permissions-Policy is configured globally. The app is embeddable via iframe by default.
POST /api/sim-data/trigger-spike has no auth guard. Any unauthenticated client can manipulate the shared simulation state. Acceptable for a demo, but the code comment only mentions multi-instance state isolation — not the unauthenticated-write risk.
localStorage stores user context including accessibility needs without sanitization on read. UserContextProvider.tsx JSON-parses the stored string with a try/catch but then spreads the result directly onto DEFAULT_CONTEXT. A tampered role field in localStorage (e.g. a crafted string value) would pass through to Gemini prompt construction unchecked by the client.
No rate limiting on Gemini-calling routes. /api/assistant, /api/alerts, and /api/briefing have no per-IP or per-session rate limit. An attacker can exhaust the Gemini API quota with repeated rapid calls. The alerts route does have an in-memory cooldown, but the assistant and briefing routes are unconstrained.

Recommended improvements

  1. Add global Next.js security headers in next.config.ts (CSP, X-Frame-Options: SAMEORIGIN, Permissions-Policy).
  2. Add server-side rate limiting middleware (e.g. upstash/ratelimit or a simple in-memory token bucket) to the three Gemini-calling routes.
  3. Re-validate the role and language values read from localStorage against the canonical allowlists before using them (same allowlists already used on the server side).
  4. Document the trigger-spike endpoint as demo-only and add a note that it should be removed or guarded before any real deployment.

Efficiency — Optimal Use of Resources

B+
Mode-scoped context injection. buildVenueSection() injects only the fields each mode requires (e.g. transit options are omitted for wayfinding), keeping prompt token counts minimal per call.
Gemini client singleton. lib/gemini.ts instantiates GoogleGenAI once per server process; subsequent calls reuse the same instance, avoiding repeated SDK initialisation overhead.
Dedup cache in alerts route. app/api/alerts/route.ts uses a 5-minute cooldown + in-memory Map to avoid re-calling Gemini for the same breach on every poll cycle. Stale entries are pruned at 2× the cooldown window, preventing unbounded memory growth.
Visibility-aware polling. CrowdDashboard.tsx skips sim-data fetches when document.visibilityState !== 'visible', preventing wasted network/CPU when the tab is backgrounded.
Concurrent inflight guard. fetchInProgressRef prevents overlapping poll requests if a previous one takes longer than the 5 s interval.
Parallel AI calls in alerts route are unbounded. app/api/alerts/route.ts fires Promise.all(newBreaches.map(...)) — if 5 gates simultaneously breach threshold, 5 Gemini calls are launched in parallel with no concurrency cap. Under high load this multiplies API cost and latency variance.
EventLog kept to 20 entries in memory. Correct for the demo lifetime, but the simEngine's module-level singleton means log entries are lost on any server restart/cold-start (no persistence). This is expected and documented but worth noting explicitly for evaluators.
Venue JSON imported at module-load time in three API routes. Each of alerts/route.ts, assistant/route.ts, and briefing/route.ts imports venue.json independently. In a serverless environment this is fine (bundled), but a shared module constant would be cleaner.

Recommended improvements

  1. Cap concurrent Gemini calls in the alerts route (e.g. process breaches in batches of 2–3 with a simple semaphore).
  2. Export a shared const venue from a single module (e.g. lib/venueData.ts) to avoid three independent JSON imports.
  3. Consider adding a Cache-Control header on /api/sim-data responses (no-store is implicit but explicit is better for CDN edge rules).

Testing — Validation of Functionality

A
127 / 127 tests pass. All 5 test suites run clean with vitest run — zero failures, zero skips.
Comprehensive unit coverage. lib/__tests__/gemini.test.ts (20 tests) covers happy path, SDK error fallback, empty response, structured parse, shape mismatch, code-fence stripping, and all priority levels. lib/__tests__/simEngine.test.ts (32 tests) covers every public function including 50-tick property-based boundary checks.
API route validation is exhaustively tested. app/api/__tests__/assistant.test.ts (32 tests) covers all 6 modes, all 3 roles, all 3 languages, all invalid body permutations, the 2000-char boundary condition, and prompt-injection in accessibilityNeeds.
Dedup / cooldown behaviour tested. app/api/__tests__/alerts.test.ts (14 tests) includes an oscillation-cooldown describe block that verifies Gemini is called exactly once for a repeated breach within the cooldown window.
Data contract tests. lib/__tests__/venueData.test.ts validates every field of venue.json — referential integrity (nearGate exists), unique IDs, coordinate shapes, and mandatory facility types. This catches data regressions before they break prompts.
No component-level tests. React components (ChatPanel, CrowdDashboard, AccessibilityPrefsModal, etc.) have zero test coverage. A React Testing Library test for ChatPanel's streaming UX and accessibility modal state machine would significantly increase confidence.
Cooldown expiry test is incomplete. The test "calls Gemini again after cooldown expires" in alerts.test.ts (line 445) ends with a comment acknowledging it cannot prove expiry without vi.setSystemTime. The assertion only re-verifies the within-cooldown case.
No integration / E2E tests. The project includes a smoke script (scripts/hello-gemini.ts) but no Playwright/Cypress suite that exercises the full fan or ops flow against a real browser.

Recommended improvements

  1. Add React Testing Library tests for ChatPanel (message render, streaming cursor, error state) and AccessibilityPrefsModal (toggle, save, skip).
  2. Complete the cooldown-expiry test using vi.useFakeTimers() / vi.setSystemTime() to advance time past ALERT_COOLDOWN_MS.
  3. Add a Playwright smoke test for the fan-page chat flow as part of CI.

Accessibility — Inclusive and Usable Design

B+
First-visit accessibility preferences modal. AccessibilityPrefsModal.tsx is a proper ARIA dialog (role="dialog", aria-modal="true", aria-labelledby). It captures mobility, vision, hearing, and sensory needs and wires them directly into Gemini prompt construction.
Live region on chat messages. The message list container has aria-live="polite" and aria-atomic="false", so screen readers announce each new assistant message without interrupting the user mid-sentence.
Thinking indicator is screen-reader friendly. The animated dots have aria-label="Assistant is thinking" on the wrapper; decorative content uses aria-hidden="true".
Semantic buttons with labels. All interactive elements (send button, language picker, role toggle, spike button) carry explicit aria-label or title attributes. The role toggle group uses role="group" + aria-label="Select role".
Keyboard navigation. Chat textarea supports Enter-to-send / Shift+Enter-for-newline; the send button is a native <button>; no custom focus traps were introduced outside the modal.
<html lang="en"> is set in src/app/layout.tsx, which is essential for screen readers and correct language detection.
Streaming cursor lacks screen-reader suppression. ChatPanel.tsx uses aria-hidden="true" on the cursor SVG and streaming dots but not on the StreamingCursor span itself. Live-region polite updates during streaming will announce partial text updates repeatedly to AT users.
Map SVG has no text alternatives. VenueMap.tsx renders an interactive SVG venue map. If the component lacks role="img" + aria-label and per-pin accessible names, keyboard and screen-reader users cannot explore the map.
Colour is the sole indicator for crowd levels. Gate cards in CrowdDashboard.tsx communicate status (low/moderate/high/critical) via colour and the text label, but no icon or pattern distinguishes states for users with colour-vision deficiency. The "high" orange and "moderate" amber are particularly close.
Focus management after modal close. AccessibilityPrefsModal.tsx does not restore focus to the trigger element after closing, which is required by WCAG 2.1 SC 2.4.3 (Focus Order).
Dark-only theme. The app uses a dark stadium colour scheme with no light-mode alternative. WCAG 1.4.6 (Enhanced Contrast) requires ≥ 7:1 for small text. Several muted grey-on-dark combinations (e.g. text-gray-600 on bg-stadium-dark) are unlikely to meet even the 4.5:1 AA threshold.

Recommended improvements

  1. Add aria-hidden="true" to the StreamingCursor component and consider pausing live-region announcements during streaming (announce only when complete).
  2. Audit SVG map pins — each pin should have a <title> child element or aria-label so screen-reader and keyboard users can access venue location data.
  3. Add distinct icons (not just colour) to each crowd-level badge (e.g. ✓, ▲, !, ⚠) to satisfy WCAG 1.4.1 (Use of Colour).
  4. Return focus to the "Preferences" button (#open-a11y-prefs-btn) when the modal closes — use a useRef to the trigger and call .focus() in closeModal().
  5. Audit contrast ratios with a tool (e.g. axe-core) — especially muted text on dark backgrounds — and adjust to meet WCAG AA (4.5:1 for body text).
