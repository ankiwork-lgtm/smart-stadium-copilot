Code Quality — Structure, Readability, Maintainability
✓ Strong

Architecture & Separation of Concerns

  • ✓ Three-layer architecture is clearly enforced: app/api/* (HTTP boundary) → lib/gemini.ts (AI reasoning) → lib/simEngine.ts (data). No cross-layer leakage. PASS
  • ✓ Single source of truth for all shared types in lib/types.ts with a JSDoc note: "Source of truth — import from here, never redefine locally." All components and API routes import from this file. PASS
  • ✓ One shared Gemini wrapper (askAssistant / askAssistantStream / askAssistantStructured) reused by every API route — no per-route Gemini duplication. 6 assistant modes are handled via a single buildPrompt() function with slot substitution. PASS
  • ✓ ChatPanel is a single parameterized component shared across Fan App and Ops Console, preventing UI duplication (design.md §6.4 decision documented). PASS

Readability & Documentation

  • ✓ Every file opens with a JSDoc block stating its purpose, the route it serves, request/response shape, and the task reference (e.g. "Task 3.1 [MUST]"). PASS
  • ✓ Constants are extracted with named symbols and inline comments: SHIFT_PROBABILITY, DELAY_PROBABILITY, RECOVERY_PROBABILITY, INCIDENT_PROBABILITY — no magic numbers. PASS
  • ✓ Section delimiters (// --- 1. Parse body ---) structure all non-trivial functions into clear numbered steps. PASS
  • ✓ Full spec-driven development chain: specs/requirements.md → specs/design.md → specs/tasks.md → implementation. Design decisions (e.g., why one shared chat component) are documented in design.md and referenced in source comments. PASS

TypeScript Discipline

  • ✓ End-to-end strict typing: LiveState, UserContext, AssistantRequest, AssistantResponse, VenueData propagate from lib/types.ts across all layers without any escape hatches in primary paths. PASS
  • ✓ Discriminated union for AssistantResponse: { success: true; text: string } | { success: false; error: true; fallbackMessage: string } — exhaustive pattern matching enforced by the compiler. PASS
  • ✓ as const used for enum-like arrays (VALID_MODES, VALID_ROLES) enabling compile-time and runtime validation from the same source. PASS

Maintainability

  • ✓ Simulation engine and AI layer are deliberately decoupled via the LiveState plain-type interface. In production the simulation layer can be replaced with real venue APIs without touching lib/gemini.ts — documented in README Q&A and design.md §5. PASS
  • ✓ resetState() exported from lib/simEngine.ts provides a clean test hook. PASS
  • ✓ ESLint configured via .eslintrc.json extending next/core-web-vitals. PASS

Security — Safe and Responsible Implementation
✓ Strong

API Key Protection

  • ✓ GEMINI_API_KEY is accessed only inside lib/gemini.ts via process.env.GEMINI_API_KEY — server-side only. The key is never imported into any "use client" component and never exposed in a NEXT_PUBLIC_ variable. PASS
  • ✓ If the key is missing, getClient() throws immediately with a clear error rather than silently passing undefined to the SDK. PASS
  • ✓ .env.example is committed; .env.local is gitignored. Reviewed README instructs cloners to copy the example before running. PASS

Input Validation & Sanitisation

  • ✓ POST /api/assistant performs explicit field-by-field validation before calling Gemini: userMessage must be a non-empty string; mode must be one of the 6 enum values; userContext.role and userContext.language are validated against allow-lists. Any deviation returns HTTP 400. PASS
  • ✓ userMessage is .trim()-ed before injection into the prompt — no raw whitespace or control characters can be injected. PASS
  • ✓ accessibilityNeeds and currentLocationHint are validated with Array.isArray and typeof === "string" checks and default to undefined if malformed — defensive defaults, not crashes. PASS

Prompt Injection Mitigation

  • ✓ System prompt and user turn are kept in separate systemInstruction and contents fields of the Gemini API call — the Gemini SDK treats these as structurally distinct, reducing prompt injection surface vs. string concatenation. PASS
  • ✓ Hard grounding rules in the system prompt include an exhaustive named-entity list of known facilities: "If the user asks about ANYTHING not on this list… respond with: 'I'm sorry, I don't have information…' Never guess, never invent." This constrains the model's output domain even if the user attempts to override instructions via the message. PASS

Response Security Headers

  • ✓ Streaming response from /api/assistant includes X-Content-Type-Options: nosniff and Cache-Control: no-cache, no-store — prevents MIME sniffing and ensures AI responses are never served from cache. PASS

Privacy & PII

  • ✓ No PII is collected. Role is a session-level UI toggle. Accessibility preferences are stored in localStorage under a namespaced key and are never transmitted to any server endpoint (only used client-side to build the userContext sent to the AI). The modal explicitly states: "These are stored locally and never shared." PASS
  • ✓ No real authentication — the role toggle is transparent session state, exactly as scoped in NFR6. No real login credentials or user data can be compromised. PASS

Efficiency — Optimal Use of Resources
✓ Strong

Streaming & Latency

  • ✓ All chat responses use generateContentStream — tokens are piped through a ReadableStream to the browser immediately. First-token latency is visible to the user within milliseconds of generation starting; no blocking full-response wait. Directly satisfies NFR1 (3–5 s target). PASS
  • ✓ Gemini connection is pre-warmed on first AppShell mount via GET /api/warmup — eliminates cold-start latency during the first real demo interaction. PASS
  • ✓ In-flight requests are tracked via AbortController in ChatPanel: if a user sends a new message while one is streaming, the previous fetch is aborted to avoid wasted tokens and concurrent stream confusion. PASS

Gemini Client — Process-Level Singleton

  • ✓ lib/gemini.ts now holds a module-level let _client: GoogleGenAI | null = null. getClient() returns the cached instance on the second and every subsequent call — the GoogleGenAI constructor and process.env read execute once per server process, not once per request. IMPROVED

Token Efficiency — Context Window Optimization

  • ✓ buildVenueSection() in lib/gemini.ts injects only the venue slice relevant to the current mode: wayfinding gets facilities + sustainability points; transport gets transit options; ops_alert/briefing get a compact summary. Not all data is injected for every call. PASS
  • ✓ buildLiveStateSection() similarly gates each data slice by mode: transit status is only injected for transport, briefing, ops_alert; the event log is only injected for briefing. Translation mode skips crowd density. PASS
  • ✓ Event log is capped at 20 entries (state.eventLog.slice(-20) in tick()), and only the last 10 are injected for briefing prompts — prevents unbounded context growth. PASS

API Call Deduplication

  • ✓ /api/alerts maintains an in-memory _seenBreachIds set — Gemini is called only once per new breach across all polling cycles. Subsequent polls return the cached alert for the same breach without re-calling the API. PASS
  • ✓ When a breach resolves, its entry is pruned from both _generatedAlerts and _seenBreachIds — so if the same breach recurs it gets a fresh AI call, but there is no memory accumulation from cleared conditions. PASS

Visibility-Aware Polling

  • ✓ CrowdDashboard now wraps its 5 s interval tick with a document.visibilityState !== "visible" guard. Polling is skipped entirely when the tab is backgrounded, and resumes immediately on visibilitychange — zero network calls and zero /api/sim-data server ticks while the user is not watching. IMPROVED
  • ✓ AlertsFeed applies the same pattern to its 8 s poll of /api/alerts. Since that route can fan-out to Gemini calls for new breaches, pausing it when hidden directly protects API quota. IMPROVED
  • ✓ Both components add a fetchInProgressRef boolean guard — if a fetch is still in-flight when the interval fires, the tick is a no-op. Prevents overlapping concurrent requests when a slow network response spans two poll intervals. IMPROVED

State & Data Architecture

  • ✓ Module-level singleton in lib/simEngine.ts means successive API calls across the server process share one in-memory state — no re-initialisation overhead per request. PASS
  • ✓ Simulation is pull-based (tick on each /api/sim-data request) rather than a real interval timer — no background threads or timers consuming server resources between requests. PASS
  • ✓ No database dependency — static venue.json and in-memory state are sufficient for demo scope; eliminates connection overhead entirely. PASS

Testing — Validation of Functionality
⚠ Partial

What Is Present

  • ✓ scripts/hello-gemini.ts — smoke test that verifies the Gemini API key is valid and the SDK can complete a round-trip call before the full app is deployed. Serves as an integration gate. PRESENT
  • ✓ scripts/test-reasoning.ts — manual reasoning/prompt test harness for iterating on prompt quality without running the full Next.js server. PRESENT
  • ✓ resetState() exported from lib/simEngine.ts provides a clean hook for programmatic test setup — allows state to be reset to a known initial condition before each test case. PRESENT
  • ✓ 4 demo flows in README function as a structured manual test script with explicit expected outcomes for each step — provides regression-check coverage for the critical user journeys. PRESENT
  • ✓ Input validation in /api/assistant is logically self-documenting: invalid modes return exact error messages (e.g. "mode must be one of: wayfinding, ops_alert…"), making manual API testing straightforward with curl or Postman. PRESENT
  • ✓ JSON parse failure path in askAssistantStructured() is covered by fallback code with a console.warn and a graceful shape-mismatch branch — demonstrates awareness of testing edge cases. PRESENT

Gaps vs. Production Standard

  • ⚠ No automated unit test suite (no Jest / Vitest configuration). resetState() is test-ready but not called from any test file. GAP
  • ⚠ No automated API integration tests for route handlers (e.g. validating that a malformed request body returns HTTP 400). GAP
  • ⚠ No component-level tests (e.g. React Testing Library for ChatPanel streaming state transitions). GAP

Recommended Additions

  • Add Vitest unit tests for lib/simEngine.ts: test tick() state transitions, triggerSpike() determinism, and resetState() idempotency.
  • Add supertest or msw-based tests for /api/assistant validation paths (400 on bad mode, 400 on empty message).
  • Add askAssistantStructured unit test: mock Gemini to return malformed JSON and assert graceful fallback is returned.
  • Promote scripts/hello-gemini.ts to a CI check by adding it to the npm test script.

Accessibility — Inclusive and Usable Design
✓ Strong

Semantic HTML & ARIA

  • ✓ Navigation bar uses <nav aria-label="Main navigation">. Page content wrapped in <main>. Landmarks are present and correctly labelled. PASS
  • ✓ Role toggle buttons use aria-pressed to communicate current selection state to screen readers. Language picker does the same. PASS
  • ✓ Accessibility modal uses role="dialog", aria-modal="true", and aria-labelledby="a11y-modal-title" — full accessible dialog pattern. PASS
  • ✓ Chat message list uses aria-live="polite" and aria-atomic="false" — streaming tokens are announced to assistive technology without interrupting the user. PASS
  • ✓ Decorative elements (emoji, avatar icons, streaming cursor) consistently marked aria-hidden="true" to prevent noise for screen reader users. PASS
  • ✓ The "thinking" indicator (ThinkingIndicator) carries aria-label="Assistant is thinking" so screen readers can convey loading state. PASS

Keyboard & Focus

  • ✓ Chat input handles Enter to send and Shift+Enter for newlines — keyboard-complete interaction without requiring a mouse click. Behaviour is stated in the visible placeholder text across all three languages. PASS
  • ✓ Send button and all modal action buttons have explicit aria-label attributes independent of visible text, ensuring screen reader announcement is unambiguous. PASS
  • ✓ Modal backdrop uses onClick to close (pointer) but the panel is inside the DOM so keyboard Tab can reach the Skip/Save buttons without trapping focus in the backdrop layer. PASS

In-Product Accessibility Features (Module D)

  • ✓ AccessibilityPrefsModal surfaces four categories: Mobility/wheelchair, Visual impairment, Hearing impairment, Sensory processing — each with a description of what guidance is adjusted. PASS
  • ✓ When any need is set, the system prompt injects: "Always prioritise accessible routes and facilities." Gate C is marked accessible: false in venue.json and the model is instructed to never recommend it to mobility-impaired users. PASS
  • ✓ Sensory Room (Gate B, East zone) is a first-class facility in venue.json and is surfaced automatically by the AI for users with sensory needs. PASS
  • ✓ Preferences are stored in localStorage and re-hydrated on every session — users with accessibility needs don't re-enter preferences on every page load. PASS
  • ✓ Multilingual support (EN/ES/FR) ensures non-English-speaking fans with accessibility needs receive accessible guidance in their own language — accessibility and inclusion compound. PASS

Visual Design for Accessibility

  • ✓ Focus states use focus:border-blue-500 focus:ring-1 focus:ring-blue-500/20 on all interactive inputs — keyboard focus is visibly indicated. PASS
  • ✓ Error state in chat uses a distinct colour class (bg-red-900/25 border border-red-500/30 text-red-300) rather than relying on icon alone — colour is supplementary, not the sole indicator. PASS
  • ✓ Disabled states on send button and textarea use disabled:opacity-40 and disabled:cursor-not-allowed — visually communicated and pointer-semantics updated. PASS
  • ✓ NFR4 satisfied: Fan-facing UI uses responsive Tailwind classes (sm:block, max-w-[85%], single-column layout) for usability on mobile-width viewports one-handed. PASS
