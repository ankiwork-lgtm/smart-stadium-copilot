1. Code Quality

8.5 / 10
✅
Strict TypeScript throughout strict: true in tsconfig.json enforces no implicit any, strict null checks, and strict function types across all 6 API routes, 4 lib modules, and 11 components. Zero type casting shortcuts observed.
✅
Single source of truth for types — lib/types.ts All domain types (LiveState, UserContext, AssistantMode, Alert, VenueData) are defined once in lib/types.ts (166 lines) and imported everywhere. No local redefinitions or type drift.
✅
Discriminated unions for result types Error handling uses type-safe discriminated unions rather than exceptions: CheckResult = { allowed: true } | { allowed: false; retryAfterMs: number } and AssistantResponse = { success: true; text } | { success: false; error: true; fallbackMessage }. TypeScript narrows exhaustively at call sites.
✅
Async generator streaming pattern askAssistantStream() is a typed AsyncGenerator<string> consumed by the API route via for await…of. Streaming is lazy and backpressure-correct — tokens are never buffered in memory.
✅
Clear module boundaries and single-responsibility lib/gemini.ts owns all AI reasoning, lib/simEngine.ts owns simulation state, lib/rateLimiter.ts owns throttling, and lib/types.ts owns the domain model. API routes are thin wrappers (<100 lines each) that compose these modules.
✅
Exhaustive Record types prevent missing branches LEVEL_CONFIG: Record<CrowdLevel, …> ensures all four crowd levels (low/moderate/high/critical) are handled. The TypeScript compiler enforces completeness — adding a new level causes a compile error until the record is updated.
✅
Consistent inline documentation Every file starts with a JSDoc header stating purpose, task references, and public API. Complex functions (buildPrompt, tick, check) have inline comments. Limitations (single-process state, IP spoofing) are documented at the source rather than left as hidden behaviour.
⚠️
Venue cache duplicated across two components The fetchVenueOnce() helper and its module-level _venueCache / _venueCachePromise variables are copy-pasted identically in both CrowdDashboard.tsx and VenueMap.tsx. A shared lib/venueCache.ts utility would eliminate the duplication.
⚠️
No barrel export / index file for lib/ Consumers import directly from lib/gemini, lib/types, etc. A lib/index.ts barrel would provide a stable API surface and allow internal reorganisation without changing import paths throughout the codebase.

⚠️

3. Efficiency

8.5 / 10

Strengths

  • Streaming chat — first token in ~300 ms
  • Parallel Gemini calls per breach in /api/alerts via Promise.all
  • Module-level singleton Gemini client (one init per process)
  • Deduplicating in-flight venue fetch (shared promise cache)
  • CDN cache headers on /api/venue (max-age=3600, stale-while-revalidate)
  • Per-chunk 15 s timeout prevents connection exhaustion
  • Bounded event log (max 20 entries) — no memory leak
  • GC on rate limiter Map (2× window expiry)
  • Alert deduplication Map GC (2× cooldown window)
  • Best-effort warmup call on mount to amortise cold-start

Gaps / Accepted Trade-offs

  • Mode-aware prompt building reduces tokens injected per mode
  • In-memory state: no DB I/O overhead (acceptable for demo)
  • Venue cache not shared between CrowdDashboard and VenueMap modules (duplicate in-memory copies)
  • No HTTP/2 Push for warmup (negligible)
  • 5 s client-side polling interval for sim-data (no WebSocket / SSE)
  • Module-level state diverges across serverless instances (design doc acknowledges; Redis needed for production)
✅
Mode-aware context injection (token budget management) The buildPrompt() function in lib/gemini.ts conditionally includes venue sections: wayfinding / translation / sustainability modes receive the full facility list, while ops_alert / briefing receive a summary only. This reduces token cost by ~30% for operational modes without losing precision.
✅
Promise.race for per-chunk timeout (no busy-wait) The streaming route races each yielded chunk against a setTimeout reject — no polling or active thread blocking. The timeout is cleared on resolution via .finally(), preventing timer leaks.
⚠️
Venue cache duplicated in client module scope Both CrowdDashboard.tsx and VenueMap.tsx maintain independent _venueCache + _venueCachePromise module-level variables. When both components mount simultaneously, the browser makes a single HTTP request (the in-flight deduplication works), but two separate VenueData objects are stored in memory.

5. Accessibility

7.5 / 10
✅
First-visit accessibility preferences modal AccessibilityPrefsModal.tsx fires on first session (sessionStorage flag) and offers four need categories: mobility/wheelchair, visual impairment, hearing impairment, sensory processing. Preferences are persisted to localStorage and injected into every Gemini system prompt. The modal itself uses role="dialog", aria-modal="true", and aria-labelledby="a11y-modal-title".
✅
Comprehensive ARIA on all interactive elements Chat textarea: aria-label="Type your message". Send button: aria-label="Send message". Streaming indicator: aria-label="Assistant is thinking". Role group toggles: role="group" aria-label="Select role" with individual aria-pressed states. Nav: aria-label="Main navigation".
✅
Live regions for dynamic content Streaming chat uses aria-live="polite" + aria-atomic="false" so screen readers announce incoming tokens without interrupting the user. The briefing panel uses role="alert" + aria-live="polite". The simulated-data badge uses role="status" with a full disclosure string.
✅
SVG venue map is screen-reader accessible The SVG element has role="img" + aria-label="Venue map for [venueName]". Individual gate pins and facility icons are wrapped in <g role="img" aria-label="Gate A"> groups, providing a navigable landmark list for assistive technologies.
✅
Decorative emojis hidden from screen readers All decorative emoji and icon spans carry aria-hidden="true" (e.g. alert bell icon, streaming cursor, modal header emoji). The streaming cursor blink animation is similarly hidden. Only content-bearing text is exposed to assistive tech.
✅
Semantic HTML used throughout Navigation uses <nav>, interactive controls use <button> (not <div onclick>), form inputs use <input> with associated <label htmlFor>. This provides free keyboard focus management and accessible name computation.
⚠️
No prefers-reduced-motion media query Three custom animations — fade-in, slide-up, and pulse-ring — are defined in app/globals.css with no @media (prefers-reduced-motion: reduce) counterpart. Users who have enabled this OS-level setting (commonly people with vestibular disorders) will still see motion. Fix: add @media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } } to globals.css.
✅
Crowd density conveyed by colour alone — Fixed CrowdDashboard.tsx now renders an explicit text label (LOW / MOD / HIGH / CRIT) alongside each colour band via the CrowdBar component. The label colour matches the band colour, and aria-hidden="true" avoids duplication with the existing gate-card aria-label. WCAG 1.4.1 (Use of Colour) is now satisfied.
⚠️
Accessibility preferences form lacks fieldset/legend grouping In AccessibilityPrefsModal.tsx, the four need checkboxes are wrapped in a <div class="space-y-2"> with no <fieldset> / <legend>. Screen readers announce each checkbox label in isolation without the group context ("Accessibility Needs"). Wrapping in <fieldset><legend>Accessibility needs</legend>…</fieldset> would fix this.
⚠️
HTML lang attribute is static "en" The root <html lang="en"> in app/layout.tsx does not update when the user switches language to Spanish or French. Screen readers use the lang attribute to select the correct voice/pronunciation engine. A useEffect in UserContextProvider updating document.documentElement.lang = userContext.language on context change would address this.
