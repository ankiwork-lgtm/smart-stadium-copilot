1. Code Quality

9.0 / 10 UPDATED
✅
Strict TypeScript with discriminated unions lib/types.ts is the single source of truth for all shared types. AssistantResponse uses a discriminated union ({ success: true; text } | { success: false; error: true; fallbackMessage }) eliminating optional-field ambiguity. CheckResult in lib/rateLimiter.ts follows the same pattern. No any casts anywhere in lib/ or app/api/.
✅
Single-responsibility modules Each file has one clear owner: lib/gemini.ts owns all Gemini I/O, lib/simEngine.ts owns simulation state, lib/rateLimiter.ts owns token-bucket logic, lib/types.ts owns shared types, lib/auth.ts owns session auth, lib/venueCache.ts owns the shared venue fetch. Route handlers in app/api/ are thin validation wrappers.
✅
Singleton client with lazy initialisation lib/gemini.ts uses a module-level _client variable and a getClient() factory. The GoogleGenAI instance is created once per process and reused across all requests — no redundant client construction on hot paths.
✅
Mode-specific token efficiency in prompt builder buildVenueSection() in lib/gemini.ts injects only the data each mode needs. ops_alert gets a facilities summary + transit; wayfinding gets full facilities + sustainability points; transport gets transit options. This avoids unnecessary token spend on irrelevant venue data.
✅
Inline JSDoc on all public API surfaces Every exported function in lib/ carries a JSDoc block describing purpose, parameters, return behaviour, and design trade-offs. Rate limiter documents multi-process limitations. SimEngine documents its singleton footnote for serverless deployments.
✅
Shared venue cache extracted to lib/venueCache.ts Fixed (v3). The previously duplicated _venueCache / _venueCachePromise / fetchVenueOnce() pattern has been extracted into a dedicated lib/venueCache.ts module. Both CrowdDashboard.tsx and VenueMap.tsx now import fetchVenueOnce from this shared module, guaranteeing a single in-memory reference and at most one network request per page load.
✅
Unused variable in alerts route resolved Fixed (v3). app/api/alerts/route.ts now uses for (const id of _seenBreachIds.keys()) instead of destructuring [id, lastAlerted]. The lint warning is eliminated.
⚠️
No barrel export for lib/ Consumers import directly from individual paths (../../../lib/gemini, ../../../lib/simEngine, etc.). A lib/index.ts barrel would clean up deep relative import paths across the app/api/ layer, though this is a style preference with no runtime impact.

2. Security

8.5 / 10
✅
Rate limiting with Retry-After on all AI-backed routes POST /api/assistant limits to 20 req/IP/min; GET /api/alerts limits to 30 req/IP/min. Both return HTTP 429 with a Retry-After header computed from the remaining window. The shared lib/rateLimiter.ts factory is independently tested.
✅
Strict enum whitelist on all user-supplied fields POST /api/assistant validates mode, userContext.role, userContext.language, and each element of accessibilityNeeds against hard-coded allow-lists before any string ever reaches Gemini. Unknown values return HTTP 400, preventing prompt injection via enum fields.
✅
Gemini API key never exposed to the client GEMINI_API_KEY is only read in lib/gemini.ts (server-side). The key is referenced in .env.example and .gitignore ensures .env.local is never committed. No client-side bundle references the key.
✅
Per-chunk stream timeout guards against stalled connections app/api/assistant/route.ts races each streamed chunk against a CHUNK_TIMEOUT_MS = 15_000 deadline via Promise.race(). A stalled Gemini stream closes gracefully with a fallback message rather than holding the HTTP connection open indefinitely.
✅
Hallucination grounding via exhaustive entity list in system prompt The system prompt injects a COMPLETE LIST of all known gates, facilities, transit options, and sustainability points with an explicit instruction: "If the user asks about ANYTHING not on this list… respond with: 'I don't have information about that facility.'" This is a defence-in-depth measure against LLM confabulation.
✅
X-Content-Type-Options: nosniff on all streamed responses The streaming response from POST /api/assistant sets X-Content-Type-Options: nosniff and Cache-Control: no-cache, no-store, preventing MIME-type sniffing and caching of per-user AI responses.
✅
Server-side authentication and role verification implemented lib/auth.ts issues HMAC-SHA256 signed session tokens stored in HttpOnly; SameSite=Strict cookies — no external dependency, uses Node.js built-in crypto. POST /api/auth/login verifies the ops PIN (constant-time comparison via timingSafeEqual) and sets the cookie. POST /api/assistant reads the verified cookie and silently downgrades any client self-declaring ops_staff without a valid session to "fan". POST /api/sim-data/trigger-spike returns HTTP 401 with no valid session and HTTP 403 for non-ops roles. The UI shows an OpsLoginModal PIN dialog; a Logout button in the nav bar clears the session via POST /api/auth/logout. GET /api/auth/session lets the shell hydrate auth state on reload. Secret configured via SESSION_SECRET; PIN via OPS_STAFF_PIN.
⚠️
In-process rate limiter does not survive multi-instance deployments The Map-based limiter is per-process. On Vercel or any horizontal scaling setup, each serverless function instance maintains its own independent counter, making the 20 req/min limit trivially bypassable by reaching different instances. A production fix requires a shared store (e.g. Upstash Redis with atomic INCR/EXPIRE).
⚠️
trigger-spike endpoint has no rate limit POST /api/sim-data/trigger-spike is the only AI-adjacent route without a rate limiter. While auth enforcement (401/403) now prevents unauthenticated abuse, an authenticated ops user could still spam the endpoint. A conservative 5 req/IP/min limiter using the existing createRateLimiter factory would close this gap.

3. Efficiency

9.0 / 10 UPDATED
✅
Token-by-token streaming for chat modes askAssistantStream() uses generateContentStream() and yields chunks via an AsyncGenerator. The route handler converts this to a ReadableStream piped directly to the HTTP response — the first token reaches the client before the full response completes.
✅
Parallel AI alert generation with Promise.all GET /api/alerts fans out all new-breach Gemini calls via await Promise.all(newBreaches.map(…)). Multiple threshold breaches generate their alerts concurrently, not serially.
✅
Alert deduplication with 5-minute cooldown window The _seenBreachIds Map prevents re-calling Gemini for the same breach on every 5-second poll. Gemini is only called on breach transitions: new breach appears or reappears after cooldown expiry.
✅
CDN cache headers on static venue data GET /api/venue returns Cache-Control: public, max-age=3600, stale-while-revalidate=86400. The venue JSON is static per session and can be served from CDN edge for a full hour, eliminating redundant origin reads on high-traffic demos.
✅
GC on rate limiter buckets prevents unbounded Map growth The gc() function in lib/rateLimiter.ts is called on every new-window reset, removing entries older than 2 × windowMs. Prevents the IP-keyed bucket Map from growing indefinitely over long server sessions with many unique IPs.
✅
Visibility-aware polling in CrowdDashboard CrowdDashboard.tsx listens to document.visibilitychange and skips fetchSimData() when the tab is hidden. Combined with an in-progress guard (fetchInProgressRef), this prevents redundant concurrent requests when the tab regains focus.
✅
Gemini warmup on first shell mount AppShell.tsx fires a best-effort GET /api/warmup on mount, pre-warming the Gemini connection so the first real user request sees minimal cold-start latency during a live demo.
✅
Shared venue cache eliminates duplicate in-memory footprint Fixed (v3). lib/venueCache.ts holds a single module-level _venueCache reference. Both CrowdDashboard and VenueMap call the same fetchVenueOnce() and receive the same object reference — zero duplicate network requests and half the previous heap footprint for the venue JSON.
⚠️
HTTP polling at 5 s rather than WebSocket push Simulation updates are fetched via setInterval(fetchSimData, 5000). For a production stadium system serving thousands of concurrent clients, a WebSocket or Server-Sent Events channel would eliminate the per-client polling overhead on the server.

5. Accessibility

9.0 / 10 UPDATED
✅
prefers-reduced-motion implemented Fixed (v3). app/globals.css now includes a @media (prefers-reduced-motion: reduce) block that sets animation-duration: 0.01ms !important, animation-iteration-count: 1 !important, and transition-duration: 0.01ms !important on all elements. The fade-in, slide-up, pulse-ring, and shimmer animations are suppressed for users with vestibular or motion-sensitivity settings. WCAG 2.3.3 (AAA) and 2.1.3 (AA) alignment achieved.
✅
Dynamic HTML lang attribute Fixed (v3). UserContextProvider.tsx now contains a useEffect(() => { document.documentElement.lang = userContext.language; }, [userContext.language]) block. When a user switches from English to Spanish or French, <html lang> is updated immediately so screen readers select the correct voice engine and pronunciation rules. WCAG 3.1.2 (Language of Parts) addressed.
✅
Accessibility preferences form uses fieldset/legend grouping Fixed (v3). In AccessibilityPrefsModal.tsx, the four need checkboxes are now wrapped in a <fieldset> with a <legend>Accessibility needs</legend>. Screen readers announce each checkbox in the context of the group. WCAG 1.3.1 (Info and Relationships) satisfied.
✅
First-visit accessibility preferences modal AccessibilityPrefsModal.tsx fires on first session (sessionStorage flag) and offers four need categories: mobility/wheelchair, visual impairment, hearing impairment, sensory processing. Preferences are persisted to localStorage and injected into every Gemini system prompt. The modal uses role="dialog", aria-modal="true", and aria-labelledby="a11y-modal-title".
✅
Comprehensive ARIA on all interactive elements Chat textarea: aria-label="Type your message". Send button: aria-label="Send message". Streaming indicator: aria-label="Assistant is thinking". Role group toggles: role="group" aria-label="Select role" with individual aria-pressed states. Language picker: role="group" aria-label="Select language". Nav: aria-label="Main navigation". OpsLoginModal PIN field: aria-describedby linked to error paragraph when present.
✅
Live regions for dynamic content Streaming chat uses aria-live="polite" + aria-atomic="false" so screen readers announce incoming tokens without interrupting the user. The briefing panel uses role="alert" + aria-live="polite". The simulated-data badge uses role="status" with a full disclosure string.
✅
SVG venue map is screen-reader accessible The SVG element has role="img" + aria-label="Venue map for [venueName]". Individual gate pins and facility icons are wrapped in <g role="img" aria-label="Gate A"> groups, providing a navigable landmark list for assistive technologies.
✅
Crowd density conveyed by colour + text label CrowdDashboard.tsx renders an explicit text label (LOW / MOD / HIGH / CRIT) alongside each colour band. The label carries aria-hidden="true" to avoid duplication with the gate-card aria-label. WCAG 1.4.1 (Use of Colour) is satisfied.
✅
Decorative emojis hidden from screen readers All decorative emoji and icon spans carry aria-hidden="true". Only content-bearing text is exposed to assistive tech.
✅
Semantic HTML used throughout Navigation uses <nav>, interactive controls use <button>, form inputs use <input> with associated <label htmlFor>. This provides free keyboard focus management and accessible name computation.
